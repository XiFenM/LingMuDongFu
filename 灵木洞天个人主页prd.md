## 2. 项目背景

当前绝大多数个人主页采用直接、线性的展示方式：顶部导航、个人介绍、项目经历、技能、博客、联系方式等。这类方式信息获取效率高，但视觉记忆点有限。

同时，3D 游戏式个人主页虽然沉浸感强，但存在明显门槛：

- 需要 3D 建模、材质、动画、物理、相机控制等复杂能力。
- 加载性能和移动端兼容性压力较大。
- 部分访客可能认为“游戏化主页”不够直接，难以快速获取信息。
- 招聘方、合作方或急需了解信息的访客可能被复杂交互劝退。

因此，本项目采用折中方案：

> 常规主页仍然保持清晰、直接、高效的信息结构，但整体视觉统一为“灵木洞天”东方仙侠美学；同时在主页中埋入一个 2D 探索小游戏入口，作为隐藏彩蛋，为愿意探索的访客提供更强的沉浸体验。

该方案兼顾实用性、审美表达和趣味性。普通访客可以快速获得关键信息；深度访客则可以通过彩蛋进入“树中仙府”，以游戏方式探索个人经历、项目、技能和博客内容。

## 3. 产品定位

### 3.1 一句话定位

一个以“灵木洞天”为美学主题的个人主页，在常规作品集网站中隐藏一个 2D 东方仙侠探索彩蛋。

### 3.2 产品核心价值

1. **信息传达清晰**  
   常规主页承担个人介绍、技能、项目、经历、博客和联系方式展示。

2. **视觉风格独特**  
   使用东方仙侠、古树、树屋、灵萤、竹简、符文、月光、云雾等元素，形成高辨识度个人品牌。

3. **互动体验有记忆点**  
   通过隐藏入口进入 2D 游戏，让访客在“灵木洞天”中探索内容。

4. **开发和维护成本可控**  
   以 2D 资产和网页动画为主，避免 3D 建模和复杂渲染。

## 4. 目标用户

### 4.1 主要用户

| 用户类型 | 典型需求 | 产品响应 |
|---|---|---|
| 招聘方 / 面试官 | 快速了解背景、技能、项目经历 | 常规主页直接展示核心信息 |
| 合作方 / 潜在客户 | 判断能力、风格和可信度 | 项目案例、技术栈、联系方式清晰呈现 |
| 技术同行 | 查看博客、项目、技术沉淀 | 博客与项目模块重点展示 |
| 朋友 / 深度访客 | 想体验有趣内容 | 通过隐藏入口进入 2D 彩蛋游戏 |
| 设计或创意相关访客 | 关注审美和表达 | 统一的灵木洞天视觉系统 |

### 4.2 用户使用场景

1. **快速浏览场景**  
   用户从简历、LinkedIn、GitHub 或社媒进入网站，希望在 1 分钟内了解站主是谁、会什么、做过什么。

2. **深度评估场景**  
   用户想进一步查看项目细节、技术博客、作品集和联系方式。

3. **探索彩蛋场景**  
   用户被页面中的特殊元素吸引，点击后进入 2D 灵木洞天小游戏，通过探索房间发现更多内容。

## 5. 产品目标

### 5.1 业务目标

- 建立具有辨识度的个人品牌。
- 提高个人主页的停留时间和记忆点。
- 在不影响信息效率的前提下增加趣味体验。
- 支持求职、技术展示、内容沉淀和社交链接转化。

### 5.2 用户目标

- 快速获取站主核心信息。
- 清楚了解技能、项目、经历、博客内容。
- 在愿意探索时获得隐藏游戏体验。
- 在视觉和交互上感受到统一的东方仙侠氛围。

### 5.3 设计目标

- 常规主页不应像游戏说明书，而应首先是一个专业、易读、可信的个人主页。
- 游戏彩蛋应隐藏但不晦涩，给细心访客以发现感。
- 视觉风格应统一，不割裂为“普通网页 + 突然出现的小游戏”。
- 所有页面内容都应服务于个人展示，而不是为了装饰而装饰。

## 6. 项目范围

### 6.1 MVP 范围

MVP 版本包含以下内容：

1. 常规个人主页首页。
2. 统一的“灵木洞天”视觉风格。
3. 基础响应式布局。
4. 项目展示模块。
5. 技能展示模块。
6. 博客或笔记入口。
7. 联系方式模块。
8. 一个隐藏的 2D 游戏入口。
9. 一个小型 2D 探索场景，至少包含 3 个可探索区域。
10. 游戏内展示部分个人信息或项目彩蛋。

### 6.2 非 MVP 范围

以下内容不进入第一版：

- 复杂战斗系统。
- 多人在线。
- 账号系统。
- 存档系统。
- 大型开放地图。
- 复杂剧情分支。
- 3D 场景。
- 重度物理交互。
- 商城或付费系统。

## 8. 信息架构

### 8.1 常规主页信息架构

#### 8.1.1 顶部导航

导航项建议：

- 首页
- 关于
- 技能
- 项目
- 博客
- 联系

可选：

- 简历下载
- GitHub
- LinkedIn

#### 8.1.2 首屏 Hero

首屏需要在 5 秒内传达：

- 我是谁。
- 我做什么。
- 我擅长什么。
- 访客下一步可以做什么。

建议内容：

- 中文名 / 英文名。
- 一句话职业定位。
- 2 到 3 个关键词标签。
- 主要 CTA：查看项目。
- 次要 CTA：阅读博客 / 下载简历 / 联系我。

视觉建议：

- 背景为巨大的灵木树心、月光、云雾、细小灵萤。
- 文字区域清晰，不被背景干扰。
- 首屏可以有轻微视差和粒子动画，但不影响阅读。

#### 8.1.3 关于我

展示内容：

- 简短自我介绍。
- 当前方向。
- 技术兴趣。
- 个人特点。
- 可选：教育背景、工作背景摘要。

视觉映射：

- 采用“洞天铭文”或“竹简卷轴”形式呈现。
- 不建议过度古风化文字，正文仍应专业、现代、易读。

#### 8.1.4 技能模块

技能模块建议设计成“灵木技能树”。

分类示例：

- AI Framework
- CUDA / GPU Computing
- PyTorch Internals
- Distributed Training
- Backend / Tooling
- Data / Statistics

展示方式：

- 树枝节点表示技能类别。
- 叶片或灵光节点表示具体技能。
- 鼠标悬浮显示简短说明。
- 点击后跳转到相关项目或博客。

#### 8.1.5 项目展示模块

项目模块是核心转化区域。

每个项目卡片包含：

- 项目名称。
- 项目类型。
- 简短描述。
- 技术栈。
- 亮点指标。
- 项目链接 / GitHub / Demo。
- 详细说明入口。

视觉映射：

- 项目卡片可设计为“灵木洞府中的法器匣”“丹方卷轴”“星盘碎片”。
- 但交互和文字必须保持清晰。

#### 8.1.6 经历 / 时间线

展示内容：

- 教育经历。
- 工作经历。
- 重要项目节点。
- 成长路线。

视觉映射：

- 使用“年轮时间线”概念。
- 每个节点像年轮上的灵光刻痕。

#### 8.1.7 博客 / 笔记

展示内容：

- 最新文章。
- 分类入口。
- 精选系列。
- 外部博客链接。

视觉映射：

- “藏书枝阁”。
- 文章卡片像竹简、书页或悬浮玉简。

#### 8.1.8 联系方式

展示内容：

- Email。
- GitHub。
- LinkedIn。
- Twitter / X。
- 小红书 / Bilibili / 其他平台，可选。
- 简历下载。

视觉映射：

- “月窗茶室”或“灵鸟传信”。
- 保持表单或链接易用。

## 9. 视觉设计规范

### 9.1 核心美学

主题名称：**灵木洞天**

关键词：

- 古树
- 树屋
- 洞府
- 仙灵
- 灵萤
- 月光
- 竹简
- 云雾
- 青苔
- 符文
- 木纹
- 纸窗
- 茶室
- 丹炉
- 剑器

整体气质：

- 东方仙侠，但不过度网游化。
- 温润、幽静、神秘，而不是高饱和玄幻特效。
- 适合技术个人主页，因此需要在“幻想感”和“专业感”之间保持平衡。

### 9.2 色彩方向

推荐主色：

| 类型 | 色彩描述 | 用途 |
|---|---|---|
| 主背景 | 深青绿 / 墨绿 / 深木色 | 页面背景、深色区域 |
| 辅助色 | 青苔绿 / 竹叶绿 | 标签、按钮、节点 |
| 高光色 | 月白 / 淡金 / 灵萤青 | 强调、CTA、发光效果 |
| 中性色 | 宣纸白 / 木灰 / 石灰 | 正文、卡片、分割线 |
| 警示色 | 暗朱砂 | 少量用于印章、重点标记 |

注意：

- 不建议大面积使用纯红、纯金、纯紫，否则容易偏页游风。
- 灵光效果应克制，避免满屏发光造成廉价感。

### 9.3 字体方向

建议采用现代字体为主，书法或古风字体为辅。

- 正文：现代无衬线字体，保证阅读效率。
- 标题：可使用带东方气质的宋体、楷体或现代标题字体。
- 装饰文字：可使用篆刻、符文、印章风格，但不可承载关键信息。

原则：

- 专业信息必须可读。
- 古风字体只用于标题、角标或装饰。

### 9.4 图形元素

常用元素：

- 年轮纹理。
- 树根曲线。
- 竹简边框。
- 纸窗格纹。
- 云纹。
- 符文环。
- 灵萤粒子。
- 月窗剪影。
- 青苔和藤蔓。

避免元素：

- 过度复杂的龙凤图案。
- 大面积金属质感。
- 高饱和页游特效。
- 过多的烟雾遮挡正文。

## 11. 隐藏 2D 游戏彩蛋需求

### 11.1 彩蛋定位

2D 游戏不是主站必经路径，而是隐藏的沉浸式作品集入口。

原则：

- 不影响常规主页信息获取。
- 不强制访客进入。
- 不让访客误以为必须玩游戏才能了解站主。
- 入口应有发现感，但不能完全不可见。

### 11.2 入口设计

可选入口方案：

#### 方案 A：灵萤入口

首屏或页面某处有一只行为略特殊的灵萤。用户悬浮或点击后，灵萤飞向树洞，出现入口提示。

优点：自然、轻盈、有发现感。  
缺点：移动端需要额外适配。

#### 方案 B：树心符文入口

在技能树或 Hero 背景中，有一个微微发光的树心符文。点击后进入游戏。

优点：主题统一，容易和灵木设定结合。  
缺点：需要避免误点。

#### 方案 C：隐藏命令入口

用户输入特定键盘序列，如 `enter the tree` 或点击 Logo 多次，进入游戏。

优点：彩蛋感强。  
缺点：发现率较低。

#### 方案 D：明确但低调入口

在页脚放置一个文案：

> “听说树心深处还有一间未开放的房间。”

点击进入游戏。

优点：不会完全错过。  
缺点：隐藏感较弱。

MVP 推荐：**方案 A + 方案 D 组合**。

- 桌面端使用灵萤入口。
- 移动端和可访问性场景使用页脚低调入口。

### 11.3 进入游戏前提示

点击入口后显示轻量弹窗：

标题：进入灵木洞天？

说明：

> 这是一个隐藏的 2D 探索彩蛋。你可以在树屋房间中查看项目、技能和笔记，也可以随时返回常规主页。

按钮：

- 进入洞天
- 还是看看主页

要求：

- 必须提供取消按钮。
- 必须说明可以随时返回。
- 游戏资源可以在用户确认后再加载。

## 12. 2D 游戏设计需求

### 12.1 游戏类型

推荐类型：

- 2D 顶视角探索。
- 或 2D 横版房间切换。

MVP 推荐：**2D 顶视角探索**。

理由：

- 更像经典 RPG 房间探索。
- 房间和走廊布局清晰。
- 易于表现树屋迷宫和洞府空间。
- 与 Phaser、PixiJS、Canvas 等技术匹配度高。

### 12.2 游戏主题

游戏场景名称：**灵木洞天**

设定：

> 访客化身为一缕误入古树的灵光，沿着树根、树心、枝廊和房间探索站主的经历、项目、技能和笔记。每个房间是一段信息的空间化表达。

### 12.3 MVP 场景结构

MVP 至少包含 3 个区域：

```text
树根玄关 → 树心中庭 → 藏书枝阁
                    └→ 炼丹小室
```

推荐完整第一版包含 5 个区域：

| 区域 | 功能映射 | 主要内容 |
|---|---|---|
| 树根玄关 | 入口 / 介绍 | 简短欢迎语、操作说明、返回主页 |
| 树心中庭 | 中心枢纽 | 通往各房间的路线、整体导航 |
| 藏书枝阁 | 博客 / 笔记 | 精选文章、学习笔记、知识库入口 |
| 炼丹小室 | 项目展示 | 重点项目、实验、Demo |
| 剑器壁龛 | 技能展示 | 技术栈、能力树、工具链 |

P1 可扩展：

| 区域 | 功能映射 | 主要内容 |
|---|---|---|
| 月窗茶室 | 联系方式 | Email、社媒、简历下载 |
| 观星树冠 | 未来计划 | Roadmap、兴趣方向、彩蛋结语 |
| 灵泉根室 | 成长时间线 | 教育、工作、重要节点 |

### 12.4 玩家控制

MVP 操作：

| 操作 | 桌面端 | 移动端 |
|---|---|---|
| 移动 | WASD / 方向键 | 虚拟摇杆 / 点击移动 |
| 交互 | E / Space / 鼠标点击 | 点击交互按钮 |
| 返回主页 | Esc / 页面按钮 | 页面按钮 |
| 打开菜单 | M / 点击图标 | 点击图标 |

要求：

- 首次进入需要显示简短操作提示。
- 所有交互对象靠近时需要有提示。
- 不要求玩家具备游戏经验。

### 12.5 游戏内交互对象

| 对象 | 功能 | 展示内容 |
|---|---|---|
| 竹简 | 打开文章或说明 | 博客、笔记、个人介绍 |
| 丹炉 | 打开项目卡片 | 项目详情、技术栈、链接 |
| 剑架 | 打开技能说明 | 技能树、能力标签 |
| 灵萤 | 引导路线 | 提示、导航 |
| 木门 | 房间切换 | 进入其他区域 |
| 石碑 | 展示经历节点 | 时间线、教育、工作 |
| 茶案 | 联系方式 | Email、社媒、简历 |

### 12.6 游戏内信息展示

游戏内不应塞入过长正文。建议采用分层信息：

1. 第一层：一句话介绍。
2. 第二层：简短卡片。
3. 第三层：跳转到常规主页对应模块或外部链接。

示例：

点击炼丹炉：

```text
丹炉中浮现一枚项目丹丸：CUDA Softmax Kernel

高性能 Softmax Forward / Backward CUDA 实现。
亮点：共享内存优化、warp-level reduction、数值稳定。

[查看项目详情] [回到房间]
```

### 12.7 返回机制

游戏界面必须始终提供明确的“返回主页”入口。

位置建议：

- 右上角常驻按钮。
- 游戏菜单中提供。
- 树根玄关处有返回门。

要求：

- 返回主页后保持用户原来的滚动位置，优先级 P1。
- 游戏不应锁死浏览器返回按钮。

## 13. 游戏美术需求

### 13.1 美术风格

建议采用：

- 2D 手绘风。
- 顶视角 RPG 风。
- 低帧率轻动画。
- 暗色木质空间 + 局部灵光。
- 东方仙侠但偏清雅，不做重度网游 UI。

### 13.2 资产清单

#### 场景 Tile / 地图资产

- 木质地板。
- 树根地面。
- 青苔地面。
- 树洞墙体。
- 木门。
- 纸窗。
- 楼梯。
- 枝廊。
- 水池 / 灵泉。

#### 装饰资产

- 竹简。
- 书架。
- 丹炉。
- 药柜。
- 剑架。
- 蒲团。
- 茶案。
- 灯笼。
- 风铃。
- 藤蔓。
- 灵萤。
- 符文。
- 小型法阵。

#### 角色资产

MVP 可选两种：

1. 不显示角色，只控制一个小灵光。  
2. 显示简单 Q 版角色或白色灵体。

推荐 MVP 使用“小灵光”作为玩家形象。

理由：

- 动画成本低。
- 不需要复杂人物设定。
- 与访客身份兼容。
- 可以用发光粒子增强表现。

#### UI 资产

- 对话框。
- 项目卡片。
- 操作提示。
- 小地图，可选。
- 返回按钮。
- 菜单面板。

### 13.3 AI 资产生成要求

由于资产可通过 AI 生成，需要制定统一规范：

- 固定视角：顶视角或 3/4 俯视角，不混用。
- 固定光源方向：建议左上或上方月光。
- 固定色彩：深青绿、木色、月白、淡金。
- 固定线条：柔和手绘，不要厚重黑边。
- 固定分辨率：Tile 资源统一 32x32、48x48 或 64x64；大型物件按比例扩展。
- 固定透明背景：角色、道具、装饰物需要透明 PNG。
- 保证同一批资产风格一致，必要时后期统一调色。

## 14. 技术方案建议

### 14.1 常规主页技术栈

推荐方案：

- Next.js / React。
- TypeScript。
- Tailwind CSS。
- Framer Motion。
- MDX 或 Markdown 博客。
- Vercel / Netlify 部署。

备选方案：

- Astro，适合内容型个人主页，性能好。
- Vite + React，结构简单，上手快。

### 14.2 2D 游戏技术栈

推荐方案：

- Phaser 3。

理由：

- 成熟的 2D Web 游戏框架。
- 支持 Tilemap、精灵动画、碰撞、相机、输入系统。
- 适合 RPG 式房间探索。

备选方案：

- PixiJS：适合自定义渲染和轻交互，但游戏系统需要自己搭。
- Canvas 原生：最轻，但开发成本会逐步上升。
- Kaboom.js：简单有趣，但长期维护和复杂场景略弱。

### 14.4 数据复用原则

常规主页和游戏彩蛋应尽量复用同一份内容数据，避免维护两套信息。

这样常规主页展示项目卡片，游戏中点击丹炉也能展示同一个项目数据。

## 22. 验收标准

### 22.1 常规主页验收

- 用户不进入游戏，也能完整了解站主信息。
- 首页首屏能明确传达身份和方向。
- 项目模块包含至少 3 个项目。
- 技能模块包含至少 5 个技能分类或核心技能。
- 联系方式可点击且有效。
- 页面在桌面和移动端均可正常浏览。
- 主题视觉统一，不出现明显风格割裂。

### 22.2 游戏彩蛋验收

- 页面中存在至少一个隐藏或半隐藏入口。
- 用户点击入口后可以选择是否进入游戏。
- 游戏资源不会阻塞常规主页首屏加载。
- 游戏中至少有 3 个可探索区域。
- 游戏中至少有 5 个可交互对象。
- 游戏中至少展示项目、技能、博客三类信息中的两类。
- 用户可以随时返回常规主页。
- 游戏在桌面端可稳定运行。

### 22.3 内容一致性验收

- 常规主页和游戏中的项目、技能信息不冲突。
- 同一项目名称、链接、技术栈保持一致。
- 游戏内文本不应替代常规主页的正式说明。

## 24. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 美术资产风格不统一 | 页面显得拼贴、廉价 | 制定 AI 生成规范，统一后期调色 |
| 游戏喧宾夺主 | 影响个人主页效率 | 游戏作为彩蛋，不作为主流程 |
| 加载过慢 | 用户流失 | 游戏资源延迟加载，图片压缩 |
| 移动端操作差 | 体验下降 | 移动端提供简化入口或触控操作 |
| 古风文案过度 | 降低专业感 | 常规主页保持现代专业文案 |
| 维护两套内容困难 | 信息不一致 | 常规主页和游戏复用同一份 JSON / MDX 数据 |
| 彩蛋发现率过低 | 游戏无人体验 | 使用灵萤入口 + 页脚低调入口双路径 |

## 28. 总结

《灵木洞天》个人主页采用“双层体验”策略：

1. **表层是专业、直接、清晰的个人主页**，满足求职、合作和快速浏览需求。
2. **里层是隐藏的 2D 探索彩蛋**，满足趣味、沉浸和个人品牌表达需求。

该方案避免了纯游戏式主页的信息阻碍，也降低了 3D 开发和建模门槛。通过统一的东方仙侠视觉系统、轻量的 2D 游戏体验和可复用的内容数据结构，可以在开发成本可控的前提下，做出具有明显差异化的个人网站。

最终目标不是把个人主页变成游戏，而是让个人主页像一座可以被发现的树中仙府：匆匆路过的人能看到门匾，愿意停留的人能走进洞天。
